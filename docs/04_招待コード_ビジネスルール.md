# 招待コード ビジネスルール（SSoT）

> **SSoT (Single Source of Truth)**: この文書が招待コードに関する唯一の正とする。
> コードの実装はこの文書に従い、矛盾が生じた場合はこの文書を優先して修正する。
> ソース: `src/services/line/process-line-event.ts` → `handleInviteCode()`
> ソース: `src/repositories/invite-codes-repo.ts` → `useInviteCode()`

---

## 1. 概要

招待コードは **admin が発行し、LINE ユーザーが LINE トーク上で送信することで、
そのユーザーを admin の管理下（account）に紐付ける仕組み**。

- 形式: `ABC-1234`（大文字3文字 + ハイフン + 数字4桁）
- 検出: 正規表現 `/^([A-Z]{3}-\d{4})$/i`（大文字・小文字どちらでもOK）

---

## 2. 基本原則

| # | ルール | 理由 |
|---|--------|------|
| R1 | **1 LINE ユーザーは 1 アカウント（admin）にのみ紐付く** | マルチテナント分離。データが混在しない |
| R2 | **初回の招待コード使用で紐付けが確定する** | 1回の操作で関係が決定する |
| R3 | **紐付け確定後、別のコード（別アカウント宛含む）を送信しても紐付けは変わらない** | 意図しないアカウント移動を防止 |
| R4 | **紐付け確定後に同じコードを再送信した場合、問診未完了なら問診を再開する** | UX: 操作に困ったユーザーの救済 |
| R5 | **コードの使用回数は `max_uses` で制限される（デフォルト=1）** | 1枚のコードで1人だけ登録 |
| R6 | **有効期限切れ・無効化されたコードは使用できない** | セキュリティ |

## 3. 全パターン分岐表

> `handleInviteCode()` の処理フロー。上から順に判定し、最初にマッチした行で処理を終了する。

| # | 条件 | 判定ステップ | BOT応答 | DB操作 |
|---|------|-------------|---------|--------|
| **P1** | ユーザーが既にコード使用済み + 問診完了 | `findInviteCodeUsageByLineUser` → あり + `intakeCompleted=1` | 「既に登録済みです。そのままご利用いただけます」 | なし |
| **P2** | ユーザーが既にコード使用済み + 問診未完了 | `findInviteCodeUsageByLineUser` → あり + `intakeCompleted=0` | 「既に登録済みです + 問診開始（質問1）」 | `mode_session` を `intake_nickname` に設定 |
| **P3** | コードが無効（DBに存在しない or status != active） | `useInviteCode` → `INVALID_CODE` | 「この招待コードは無効です」 | なし |
| **P4** | コードが有効期限切れ | `useInviteCode` → `CODE_EXPIRED` | 「有効期限切れです」 | `invite_codes.status` → `expired` |
| **P5** | コードが使用上限到達 | `useInviteCode` → `CODE_EXHAUSTED` | 「使用上限に達しています」 | なし |
| **P6** | 同一ユーザーが同じコードを二重使用 | `useInviteCode` → `ALREADY_USED` | 問診状態に応じてP1/P2相当の応答（通常はP1/P2で先にキャッチ） | 問診未完了なら `mode_session` 設定 |
| **P7** | 同一ユーザーが別コード・同一アカウント宛 | `useInviteCode` → `ALREADY_USED` | 問診状態に応じてP1/P2相当の応答（通常はP1/P2で先にキャッチ） | 問診未完了なら `mode_session` 設定 |
| **P7b** | **同一ユーザーが別コード・別アカウント宛** | `useInviteCode` → `ALREADY_BOUND` | 問診状態に応じてP1/P2相当の応答（R3: 紐付け変更不可） | 問診未完了なら `mode_session` 設定 |
| **P8** | **別ユーザーが他人の使用済みコード（max_uses=1）を入力** | `useInviteCode` → `CODE_EXHAUSTED` | 「使用上限に達しています」 | なし |
| **P9** | 新規ユーザーが未使用の有効コードを入力 | `useInviteCode` → `success` | 「登録完了 + 問診開始（質問1）」 | usage作成、use_count++、紐付け、service_status作成 |

## 4. フローチャート

```
ユーザーが "ABC-1234" を送信
│
├─ 1. findInviteCodeUsageByLineUser(lineUserId)
│   │
│   ├─ [使用履歴あり] ── このユーザーは過去にコードを使ったことがある
│   │   │
│   │   ├─ checkServiceAccess → intakeCompleted?
│   │   │   ├─ [YES] → reply: 「既に登録済み。そのままご利用ください」 (P1)
│   │   │   └─ [NO]  → reply: 「登録済み」+「問診 質問1」を同時送信 (P2)
│   │   │              DB: mode_session → intake_nickname
│   │   └─ return
│   │
│   └─ [使用履歴なし] ── 初めてコードを使うユーザー
│       │
│       └─ 2. useInviteCode(code, lineUserId)
│           │
│           ├─ INVALID_CODE  → reply: 「無効なコードです」 (P3)
│           ├─ CODE_EXPIRED  → reply: 「期限切れです」 (P4)
│           ├─ CODE_EXHAUSTED → reply: 「使用上限です」 (P5)
│           ├─ ALREADY_USED  → 問診状態に応じてP1/P2相当 (P6/P7)
│           ├─ ALREADY_BOUND → 問診状態に応じてP1/P2相当 (P7b, R3)
│           │
│           └─ SUCCESS → 紐付け処理
│               ├─ user_accounts 紐付け
│               ├─ user_service_statuses 作成
│               ├─ mode_session → intake_nickname
│               └─ reply: 「登録完了」+「問診 質問1」を同時送信 (P9)
```

## 5. テーブル構造（関連部分）

### invite_codes
| カラム | 型 | 説明 |
|--------|---|------|
| id | TEXT PK | UUID |
| code | TEXT UNIQUE | `ABC-1234` 形式 |
| account_id | TEXT FK | 紐付け先アカウント |
| created_by | TEXT FK | 発行者（account_membership.id） |
| max_uses | INTEGER | 使用上限（デフォルト=1） |
| use_count | INTEGER | 現在の使用回数 |
| status | TEXT | `active` / `expired` / `revoked` |
| expires_at | TEXT | 有効期限（ISO8601, nullable） |

### invite_code_usages
| カラム | 型 | 説明 |
|--------|---|------|
| id | TEXT PK | UUID |
| invite_code_id | TEXT FK | 使用されたコード |
| line_user_id | TEXT | 使用した LINE ユーザー |
| used_at | TEXT | 使用日時 |

**UNIQUE制約**: `(invite_code_id, line_user_id)` — 同一ユーザーが同じコードを2回使用できない

## 6. エッジケースと対処方針

### Q: ユーザーを別の admin に移動したい場合は？
**A**: 現行設計では LINE トーク上からの移動は不可（R3）。
管理画面から `user_accounts.client_account_id` を直接変更する管理者操作が必要。
将来的には admin 間のユーザー移動機能を実装予定。

### Q: max_uses > 1 のコードで複数人が登録する場合は？
**A**: 1つのコードを複数のユーザーが使用可能。
各ユーザーごとに `invite_code_usages` レコードが作成される。
`use_count >= max_uses` に達したら新規ユーザーは `CODE_EXHAUSTED` になる。

### Q: admin が間違えてコードを revoke した場合は？
**A**: 既に使用済みのユーザーの紐付けには影響しない。
新規ユーザーがそのコードを使おうとすると `INVALID_CODE` になる。
