# 画像解析・記録反映確認表

> 最終更新: 2026-03-12  
> 対象: diet-bot v1.0.0 (Phase 1 MVP)

---

## 1. 画像処理パイプライン全体図

```
[ユーザー] LINE画像送信
    │
    ▼ (1) 同期処理 — handleImageMessageEvent
    ├─ LINE Content API → 画像バイナリ取得
    ├─ R2ストレージ保存 (intake/{userAccountId}/{date}-{messageId}.jpg)
    ├─ conversation_messages (type=image) 作成
    ├─ message_attachments 作成
    ├─ image_analysis_jobs 作成 (status=pending, provider=openai_vision)
    ├─ LINE_EVENTS_QUEUE.send() → Queue投入
    └─ 即時返信「📷 解析中です...」
    
    │
    ▼ (2) 非同期処理 — Queue Consumer (lineQueueConsumer)
    ├─ R2.get() → 画像バイナリ取得
    ├─ Base64変換
    ├─ [Step 1] 分類 — OpenAI Vision API (CLASSIFY_PROMPT)
    │   └─ → { category, confidence }
    ├─ [Step 2] 詳細解析 — カテゴリ別プロンプト
    │   ├─ meal_photo → MEAL_ANALYSIS_PROMPT
    │   ├─ nutrition_label → NUTRITION_LABEL_PROMPT
    │   ├─ body_scale → SCALE_PROMPT
    │   ├─ progress_body_photo → (コメントのみ)
    │   └─ other / unknown → (コメントのみ)
    ├─ image_intake_results INSERT (applied_flag=0)
    ├─ bot_mode_sessions → current_step='pending_image_confirm'
    └─ LINE Push通知 → 解析結果 + [確定/取消]
    
    │
    ▼ (3) ユーザー確認 — pending_image_confirm
    ├─ 「確定」→ handleImageConfirm → メインテーブル反映 (applied_flag=1)
    ├─ 「取消」→ handleImageDiscard → 破棄 (applied_flag=2)
    └─ 24h無応答 → Cronで自動破棄 (applied_flag=3)
```

---

## 2. 画像分類（カテゴリ別）

| # | カテゴリ | image_category | 分類条件 | 信頼度閾値 |
|---|----------|----------------|----------|-----------|
| 1 | 食事写真 | `meal_photo` | 食べ物・料理が写っている | confidence ≥ 0.7 |
| 2 | 栄養成分表示 | `nutrition_label` | パッケージの栄養表示ラベル | confidence ≥ 0.8 |
| 3 | 食品パッケージ | `food_package` | 食品のパッケージ（コンビニ弁当等） | confidence ≥ 0.7 |
| 4 | 体重計 | `body_scale` | 体重計の数値表示 | confidence ≥ 0.8 |
| 5 | 体型写真 | `progress_body_photo` | 全身・半身の体型写真 | confidence ≥ 0.7 |
| 6 | その他 | `other` | 食事/体重/体型に関連するがカテゴリ不明 | - |
| 7 | 不明 | `unknown` | 関連性が判定できない | - |

## 3. カテゴリ別 抽出データ・DB保存先・ユーザー返信

### 3.1 食事写真 (meal_photo)

| 項目 | 抽出データ | 型 |
|------|-----------|-----|
| 料理名 | items[].name | string |
| 推定カロリー | items[].estimatedCalories | number (kcal) |
| 推定重量 | items[].estimatedWeight | number (g) |
| 合計カロリー | totalCalories | number (kcal) |
| たんぱく質 | protein | number (g) |
| 脂質 | fat | number (g) |
| 炭水化物 | carbs | number (g) |
| 食事区分 | mealType | breakfast/lunch/dinner/snack/other |
| コメント | comment | string (80文字以内) |

**DB保存先 (確定時):**
| テーブル | カラム | データ |
|----------|--------|--------|
| `daily_logs` | id, log_date | 今日のログ (ensureDailyLog) |
| `meal_entries` | meal_type, meal_text, calories_kcal, protein_g, fat_g, carbs_g, confirmation_status='confirmed' | 栄養データ |
| `image_intake_results` | extracted_json, proposed_action_json, applied_flag=1, confirmed_at | 解析結果 |

**ユーザー返信例:**
```
🍽️ 食事を記録しました！

・チキンカレー（約650kcal）
・サラダ（約80kcal）

合計: 約730kcal
P: 35g / F: 20g / C: 95g

バランスの良い食事です。
[✅ 確定] [❌ 取消]
```

**管理画面反映:**
- LINEユーザー管理 → ユーザー詳細 → 「食事記録」タブ → 日付別食事一覧
- ダッシュボード → 「今日の記録数」カウント

**LIFFダッシュボード反映:**
- ホーム → 今日のサマリー（カロリー, P/F/C）
- ホーム → 今日の食事リスト
- 記録タブ → 過去の記録一覧

## 4. proposed_action_json の構造

画像解析結果に基づき、「確定」時にどのテーブルに何を書き込むかを定義するJSON。

### 4.1 食事記録の場合
```json
{
  "action": "create_or_update_meal_entry",
  "user_account_id": "ua_xxx",
  "client_account_id": "acc_client_xxx",
  "meal_type": "lunch",
  "meal_text": "チキンカレー、サラダ",
  "calories_kcal": 730,
  "protein_g": 35,
  "fat_g": 20,
  "carbs_g": 95
}
```

### 4.2 体重記録の場合
```json
{
  "action": "upsert_weight",
  "user_account_id": "ua_xxx",
  "client_account_id": "acc_client_xxx",
  "weight_kg": 72.5
}
```

### 4.3 体型写真の場合
```json
{
  "action": "create_progress_photo",
  "user_account_id": "ua_xxx",
  "client_account_id": "acc_client_xxx",
  "r2_key": "intake/ua_xxx/2026-03-12-msg123.jpg",
  "photo_date": "2026-03-12"
}
```

### 4.4 記録不要の場合
```json
{
  "action": "none"
}
```

## 5. OpenAI Vision API プロンプト一覧

| プロンプト | 用途 | モデル | max_tokens |
|-----------|------|--------|-----------|
| CLASSIFY_PROMPT | 画像分類 (7カテゴリ) | gpt-4o | 200 |
| MEAL_ANALYSIS_PROMPT | 食事詳細解析 (カロリー, PFC) | gpt-4o | 800 |
| NUTRITION_LABEL_PROMPT | 栄養ラベル読取 | gpt-4o | 500 |
| SCALE_PROMPT | 体重計数値読取 | gpt-4o | 200 |
| (analyzeImage systemPrompt) | 統合解析 (image-analyzer.ts) | gpt-4o | 800 |

> **注意**: image-analysis.ts (Queue Consumer) と image-analyzer.ts の2箇所に分散してプロンプトが定義されている。将来的には統一が望ましい。

## 6. エラーハンドリング

| エラー | 原因 | 対応 |
|--------|------|------|
| LINE Content API 失敗 | トークン期限切れ, 画像削除済み | 「画像の取得に失敗しました。もう一度お試しください。」 |
| R2 保存失敗 | ストレージエラー | 「画像の保存に失敗しました。しばらくしてから再度お試しください。」 |
| Queue 投入失敗 | キューサービスエラー | ログのみ（DBにジョブが残るため致命的ではない） |
| OpenAI API 失敗 | API制限, ネットワーク | image_analysis_jobs.status='failed', ユーザーにエラー通知 |
| JSON パース失敗 | AI応答フォーマット異常 | category='unknown' として処理 |
| 確認タイムアウト | 24h無応答 | Cronジョブで applied_flag=3 (expired) に自動更新 |

```
  └── intake/
       └── {userAccountId}/
            ├── 2026-03-12-{lineMessageId}.jpg
            ├── 2026-03-12-{lineMessageId}.jpg
            └── ...
```

| 設定 | 値 |
|------|-----|
| バケット名 | diet-bot-media |
| キー形式 | `intake/{userAccountId}/{date}-{messageId}.jpg` |
| Content-Type | image/jpeg (LINE Content APIから取得) |
| アクセス | R2バインディング経由のみ（パブリックURLなし） |
